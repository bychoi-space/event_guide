## 2. 프로젝트 가이드 파일의 역할 (Roles)
본 프로젝트의 산출물은 세련된 에디토리얼 디자인을 담은 시스템으로 구성되어 있으며, 각 파일의 역할은 다음과 같습니다.

*   **[rules.md](file:///c:/event_guide/rules.md) (본 파일):** 프로젝트의 전체적인 방향성, 원칙, 그리고 업무 룰과 역할 분담을 기록한 마스터 통제 규칙입니다.
*   **[design.md](file:///c:/event_guide/design.md) (디자인 가이드 토큰):** Cohere의 브랜딩 아이덴티티(Canvas White, Near-Black, Enterprise Green, Coral 등)와 타이포그래피 스케일, 둥근 모서리 반경 규칙 등을 명시한 프론트엔드 스타일 설계 가이드입니다.
*   **[style.css](file:///c:/event_guide/style.css) (마스터 스타일시트):** `design.md`를 표준화하여 제작된 실질적인 웹 스타일시트입니다. 웅장한 디스플레이 서체와 미니멀한 규칙선을 연출합니다.
*   **[list.html](file:///c:/event_guide/list.html) (카드 한눈에 보기):** NEW템플릿에 구성된 모든 카드(39종 중 핵심 14종 우선 배치)를 유형별(배너/상품/프로모션/내비게이션)로 분류하여 직관적인 컴포넌트 리스트 타입으로 제시하는 포털 페이지입니다.
*   **[detail.html](file:///c:/event_guide/detail.html) (인터랙티브 상세 가이드):** 각 카드별 Front-end 화면 레이아웃 작동 방식, Back-end(NBOS)에서의 데이터셋 필수 설정, 주의사항 가이드를 사이드바 트리 조작을 통해 실시간으로 조립형 탐색을 제공하는 고기능성 상세 페이지입니다.
*   **[wireframes.json](file:///c:/event_guide/wireframes.json) 및 `db/wireframes/wf_*.js` (와이어프레임 데이터베이스):** 가이드 내에 동적으로 주입되는 모든 카드들의 프론트엔드 모듈 마크업 구조와 레이아웃 와이어프레임 HTML 템플릿들을 담고 있는 정보 파일들입니다. 기존 단일 파일 구조에서 에디팅 속도 향상을 위해 39개의 개별 모듈 단위 파일로 분리되었습니다.

## 4. Local 실행 및 CORS 보안 유의사항 (중요)
- **현상**: 로컬 환경(`file:///C:/...`)에서 브라우저로 `detail.html`을 즉시 더블클릭하여 기동할 경우, 브라우저 자체의 강력한 보안 정책으로 인해 비동기 JSON 로드(`fetch`), 이미지/리소스 외부 주소 로딩, 그리고 주소창 조작 및 세션 이력 갱신(`window.history.pushState` / `replaceState`) 시 **CORS 보안 차단 에러 및 고유 보안 Origin(Unique Security Origin) 차단 경고**가 빈번히 발생할 수 있습니다.
- **개발 필수 수칙 (CORS 및 로컬 파일 보안 정책 감안)**:
  * **반드시 로컬 파일 보안 정책(CORS Origin 정책)을 100% 감안하여 시스템을 기획 및 코딩해야 합니다.**
  * 개발 시 비동기 `fetch` 호출에만 전적으로 의존하는 구조를 절대 지양하며, 로컬 환경에서 정적 자산 로드가 불가능할 때를 대비해 스크립트 기반 데이터 주입 주입 방식(`window.ExhibitionGuides` 등 전역 네임스페이스 바인딩)을 상시 유지하고 결합해야 합니다.
  * 모든 도메인 간 리소스 참조는 CORS 제약 없이 로컬 static 환경(`file:///` 프로토콜)에서도 에러나 경고 없이 즉각 로드 및 구동되는 구조여야 함을 개발의 대전제로 삼습니다.
- **안전 규칙 및 구현 장치**:
  1. **Dual-Bypass 설계**: 모든 데이터 로드(`fetch` API 등) 시, 로컬 파일 환경에서 보안 차단 에러가 나거나 네트워크가 분리되더라도 오프라인 무중단 서비스가 가능하도록 **완벽하게 동기화된 인라인 로컬 백업 데이터셋**으로 즉시 자동 예외 전환(`try-catch` 및 fallback)되도록 이중 방어 메커니즘을 상시 구축해야 합니다.
  2. **환경 감지 및 브라우저 API 호출 제어**: 로컬 파일 시스템 구동 여부를 판별하여, `file:` 프로토콜 기반일 때에는 브라우저가 보안 위협으로 판단하는 `history.replaceState` 또는 `pushState` API 호출을 사전에 원천 우회(Bypass) 및 스킵하여 개발자 도구 콘솔에 어떠한 에러/경고 로그도 남지 않게 해야 합니다.
  3. **이미지 로딩 예외 방어 (onerror 처리)**: 외부 도메인(`img.lfmall.co.kr` 등)에서 이미지를 로드하는 경우 로컬 환경이나 오프라인 환경에서 네트워크 차단으로 인한 엑스박스(이미지 깨짐) 현상을 미연에 예방하기 위해, 모든 원천 리소스 이미지에는 인라인 SVG 데이터 URI 형태의 디폴트 Fallback(`onerror="this.src='data:image/svg+xml;...'"`) 처리를 명세화하여 레이아웃 무결성을 유지합니다.
  4. **보안 origin 격리 고려**: 아이프레임(Iframe) 또는 프레임 구조 접근 시에도 `file:` URLs가 unique security origin으로 격리되어 동작하는 점을 깊이 인지하고, 프레임 간 데이터 접근이나 도메인 종속적 윈도우 조작 코드를 상시 지양하고 독립적으로 동작할 수 있도록 설계해야 합니다.


## 5. 향후 카드별 분석 고도화 작업 체크리스트 (Checklist Rules)
본 시스템은 어떠한 서버 환경 없이도 오프라인 로컬 환경에서 100% 동작해야 하므로, 향후 39종 전체 카드를 점진적으로 분석/이식할 때 반드시 아래 **체크리스트** 및 규정을 준수하십시오.

1. **원천 소스코드 분석 프로세스 규칙**:
   - **Front-end (`lf-ui`)**: `src/mobile/lfmall/components/planning/planningTemplate/` 디렉토리 하위의 React/TypeScript 소스 파일을 추적하여 CSS Class 토큰, 디폴트 폰트 사이즈, Swiper/반응형 동작, 이미지 비율, 텍스트 글자 수 입력 조건(예: 한글 10자 등)을 정밀 분석합니다.
   - **NBOS (`lfmall-admin-renew`)**: `src/main/webapp/lfmadm/Base/Popup/` 내 카드별 `frmPlanCard*.xfdl.js` 어드민 번역본 소스 파일을 참조하여 넥사크로 DataSet(`dsPlanCardInfo`, `dsCardDetail`)의 고유 Column 명칭, 콤보 박스 코드값 리스트, 저장 로직 바인딩 관계를 1:1로 추출해 상세 가이드에 명시합니다.

2. **데이터베이스 무결성 및 동기화**:
   - 신규 분석된 카드 데이터는 반드시 `db.js` 내의 `window.ExhibitionGuides`와 `window.ExhibitionWireframes` 객체 안에 동시에 반영하십시오.
   - CORS 차단 방어용으로 설계된 `detail.html` 내부의 `loadLocalDatabaseDirectly()` 예외 블록 변수들과 `db.js` 마스터 데이터가 완벽하게 일치하는지 무결성 상태를 유지해야 합니다.

3. **자바스크립트 템플릿 리터럴 이스케이프 주의 (★치명적)**:
   - `detail.html` 내에서 템플릿 문자열을 사용해 HTML 마크업을 동적 렌더링할 때, `${variable}` 형태의 변수 삽입 구간 앞에 백슬래시(`\`)를 붙이지 마십시오. (예: `\${data.name}`이 아닌 `${data.name}`으로 표기)
   - 만일 이스케이프가 가미될 경우, 브라우저가 연산 처리를 누락하고 화면에 변수명 자체 `${...}`를 그대로 텍스트 출력하는 기하학적 렌더링 깨짐이 유발됩니다.

4. **모바일 와이어프레임 프리미엄 디자인 토큰 유지**:
   - 모바일 UI 프레임 내부 와이어프레임 작성 시, plain한 원색 사용을 철저히 금지합니다.
   - `style.css`에 선언된 Cohere 프리미엄 디자인 시스템 토큰(`var(--near-black)`, `var(--canvas-white)`, `var(--coral)`, `var(--soft-stone)`, `var(--pale-green)`, `var(--enterprise-green)`)만을 조합해 극도의 미니멀리즘과 세련된 레이아웃을 통일감 있게 유지해야 합니다.

5. **가이드 페이지 통일화 및 미니멀리즘 설계 규칙 (★기획 관리 통제 원칙)**:
   - **디바이스 프레임 일관성**: 브라우저(PC) 및 스마트폰(Mobile) 시뮬레이션 프레임의 테두리(Border) 두께는 복잡도를 배제하고 미니멀한 균형감을 선사하기 위해 **무조건 `1px` 두께의 완전 직각(모서리 라운드 `0px` 제거) 사각형 구조**로 완전 통일합니다.
   - **제목 작명 규칙 통일**: 듀얼 미리보기의 헤드라인 레이블은 잡음(이모지, 복잡한 부가텍스트)을 완전히 제거하고 반드시 **`"PC 미리보기"`**, **`"Mobile 미리보기"`**로 대칭 규칙을 완벽하게 통일하여 작성합니다.
   - **컴포넌트 비주얼 통일감**: 개별 와이어프레임 렌더링 시, 특정 카드는 룩북 실물 사진을 보여주고 다른 카드는 회색 점선박스를 보여주는 식의 불일치를 엄격히 금지합니다. 모든 카드 프리뷰는 **실제 상용 서비스 퀄리티의 세련된 고화질 패션 룩북 이미지 배경, 정밀한 폰트 타이포그래피 정렬(중앙 정렬 중심), 그리고 단색 라운드 버튼 토큰으로 통일**하여 고급스러운 심미성을 영구 유지합니다.
   - **불필요 오브젝트 배제**: 시각적 노이즈를 일으키는 개발용 카드 코드 배지(예: `[MD12]` 뱃지 등)나 중언부언하는 장황한 설명 텍스트를 최소화하여 MD 관점의 극대화된 가독성을 보장합니다.

      *   **이미지 업로드 가이드 필수화**: [이미지 업로드] 기능이 존재하는 모든 카드(예: 탑배너, 배너 등)에는 반드시 상세페이지 상단에 이미지 업로드 규격(PC/MO 권장 사이즈, 허용 파일 포맷) 및 파일당 최대 용량 제한 가이드(`imageGuidelines`)를 필수적으로 분석하여 작성 및 명시해야 합니다.

      *   **LFmall 실물 UI 극대 대칭성 준수 (★가장 강력한 시각 규정)**: 앞으로 모든 Front UI 와이어프레임 미리보기와 레이아웃을 그릴 때, 대충 dummy 박스나 원색 회색으로만 렌더링하는 행위를 엄격히 금지합니다. 실물 캡처 이미지 사양(예: 탑배너의 Dimmed, 구매혜택의 절취선/notch 쿠폰 티켓, **유의사항 카드의 gray 아코디언 토글 헤더 및 토글 화살표 `^`/`v` 접고 펼치기 인터랙션** 등)을 완벽하게 분석하여 캡처 이미지와 극도로 대칭되고 고급스러운 상용 퀄리티로 렌더링하여야 합니다.

  6. **LFmall 브랜드 디자인 규격 및 1:1 대칭 미리보기 원칙 (★치명적 품질 통제 룰)**:
     - **원천 CSS 및 디자인 1:1 이식**: 미리보기(`wireframeHtml`) 구현 시 임의의 상상력을 배제하고, 원천 React 마크업 구조와 CSS 스타일(폰트 두께, 크기, 행간, 자간, 컬러값, 패딩/마진 여백 및 정렬선) 또는 제공된 디자인/캡처 이미지를 픽셀 단위로 철저히 분석하여 1:1로 완벽히 동일하게 구현합니다.
     - **LFmall 디자인 표준 자동 적용**: 구체적인 소스나 이미지가 부족한 영역은 아래 LFmall 표준 규격을 준수하여 작성합니다.
       * **Typography (서체 & 자간)**: `font-family: "Malgun Gothic", "Apple SD Gothic Neo", sans-serif;`를 모바일 뷰 기본값으로 설정하고, 국문 가독성을 위해 자간 `letter-spacing: -0.3px` ~ `-0.8px`, 줄간격 `line-height: 1.3` ~ `1.5` 범위를 적용합니다.
       * **Color & Border System (색상 및 외곽선)**: LFmall 시그니처 딥 블랙 `#000000`, 묵직한 카콜 그레이 `#111111`을 메인 텍스트 및 경계선에 사용하며, 포인트 브랜드 컬러는 코랄 오렌지 `#e25822`(또는 `#ff4b2b`), 경계선은 실버 그레이 `#e0e0e0` 또는 `#d2d2d2`를 적용합니다. 기획전 메인 카드 프레임은 직각(`border-radius: 0;`)으로 하고 내부 서브 요소에 한해 `2px` ~ `4px` 수준의 둥글기를 적용합니다.
       * **Component Layout (컴포넌트 레이아웃)**: 카드 내부 콘텐츠의 좌우 기본 여백은 `20px`로 맞추고, 검색창 카드는 좌측 정렬 및 2px 블랙 보더 직각 사각형 구조를 사용합니다. 결제 혜택 카드는 접고 펼치는 아코디언 토글 헤더와 함께 둥근 모서리(`border-radius: 6px`)에 1:1.5 비율의 수직형 그라데이션 신용카드 디자인을 적용합니다.
       * **Interaction (상호작용)**: 모든 앵커 태그(`<a>`)와 버튼은 브라우저 기본 포커스 아웃라인 및 밑줄을 제거하고 `cursor: pointer;` 스타일을 부여합니다. 가로형 스크롤 바는 감추고(`::-webkit-scrollbar { display: none; }`), 모바일 스와이프를 위해 `overflow-x: auto; white-space: nowrap;` 스타일을 필수 적용합니다.


  7. **미리보기 Mockup 내 가로폭 및 크기 제약 준수 규칙 (★레이아웃 깨짐 원천 차단)**:
     - **Mockup 컨테이너 가로폭 제약**: 가이드 상세페이지의 PC 미리보기 영역(`.desktop-mockup .browser-content`) 및 Mobile 미리보기 영역(`.phone-mockup .phone-screen`)은 가로폭이 반응형으로 좁아질 수 있습니다.
     - **임의의 대형 max-width 지정 금지**: 와이어프레임 마크업 내부의 그리드나 컨테이너(`grid-container`, `tile-item` 등)에 `max-width: 1080px`, `max-width: 220px`와 같은 과도한 고정 또는 대형 최대 너비를 지정하면, Mockup 영역 내에서 유연하게 축소되지 못해 그리드가 바깥으로 튀어나가거나 렌더링이 깨집니다.
     - **반응형 % 비율 및 aspect-ratio 사용**: Mockup의 가로 비율에 맞춰 자연스럽게 축소될 수 있도록 `.desktop-mockup` 전용 CSS 스코프 내에서 `max-width: 100%`, `height: auto`, `aspect-ratio: 1/1` 등의 상대 비율과 유연한 제약 조건을 지정하여 모바일 및 PC 뷰포트 내 레이아웃 무결성을 영구 보장하십시오.


   8. **브라우저 직접 검증 생략 원칙 (★개발 효율성 극대화)**:
      - **에이전트 브라우저 검증 생략**: 개발 및 분석 생산성 향상을 위해 AI 에이전트가 브라우저(Browser Subagent)를 직접 기동하여 `detail.html`의 렌더링이나 인터랙션을 실시간으로 직접 확인 및 촬영하는 검증 단계는 전면 생략합니다.
      - **사용자 직접 검증**: 와이어프레임 마크업 및 데이터베이스 수정이 완료되면 최종 가이드 결과물만 안전하게 갱신하여 제출하고, 브라우저 상의 세부적인 기능 동작 및 UI 레이아웃 검증은 사용자가 직접 수행합니다. 에이전트는 코드 무결성 및 JSON 문법의 구조적 유효성 검증에 집중합니다.


## 6. 완벽한 화면 미리보기(Mockup) 분석 및 검증 규정
본 가이드 시스템의 핵심인 화면 미리보기(PC & Mobile)의 완벽성을 보장하기 위해 아래 3단계 분석 프로세스 및 5대 자가 검증 체크리스트를 준수하여 구현을 수행해야 합니다.

### 1) 화면 미리보기 구현을 위한 3단계 분석 프로세스
1. **1단계: React 마크업 구조 분석 (Structure)**
   - `lf-ui` 내 해당 카드의 React 컴포넌트 소스(`.tsx`)를 추적하여 최외곽 컨테이너부터 하위 텍스트/이미지 노드까지의 HTML DOM 트리를 모사합니다.
   - 실제 소스에서 제공되는 CSS 클래스명과 구조적 위계를 그대로 와이어프레임 마크업에 이식합니다.
2. **2단계: CSS 스타일 및 반응형 분석 (Style)**
   - 컴포넌트 전용 CSS/SCSS 스타일시트를 역추적하여 폰트 규격(크기, 줄간격, 자간), 반응형 분기점(PC 4열 vs Mobile Swiper 가로 스크롤 등), 주요 브랜드 테두리선 및 컬러 토큰 정보를 완벽하게 추출하여 미리보기에 녹여냅니다.
3. **3단계: Mockup 고유 제약 및 데이터 매핑 (Constraint & Data)**
   - PC/MO 미리보기 뷰포트 내 가로폭 제약을 감안하여 상대 너비 비율(`width: 100%`) 및 `aspect-ratio`를 부여하고, 더미 대신 LFmall 실물 상품 데이터와 패션 룩북 고화질 이미지를 템플릿 변수에 매핑합니다.

### 2) 완벽한 미리보기를 위한 자가 검증 체크리스트 (5대 규칙)
- [ ] **뷰포트 격리성**: PC Mockup과 Mobile Mockup(360px 내외) 환경 모두에서 강제 가로 스크롤이나 요소 튀어나옴 없이 완벽한 반응형 렌더링을 제공하는가?
- [ ] **타이포 및 자간 완벽성**: LFmall 표준 타이포 규격(Malgun Gothic, 자간 `-0.3px` ~ `-0.8px`, 줄간격 `1.3` ~ `1.5`)이 컴포넌트 전반에 정밀하게 적용되었는가?
- [ ] **이미지 비율 및 왜곡 방지**: 이미지 원본 해상도 비율이 깨지지 않도록 `object-fit: cover` 및 상대 비율 조건이 알맞게 부여되었는가?
- [ ] **인터랙션 미니멀 구현**: Swiper 슬라이드 이동, 아코디언 토글, 카테고리 탭 전환 등 미리보기 상에서 기능적으로 필요한 필수 스크립트 동작이 바닐라 JS로 연동되는가?
- [ ] **시각적 완성도**: 단순 회색 박스를 지양하고, 실제 패션 룩북, 코랄 Tag 배지, 정밀 제작된 SVG 아이콘이 적재적소에 결합되어 상용 사이트 수준의 퀄리티를 유지하는가?

### 3) Figma 없이 분석하는 '프리미엄 Front UI 미리보기' 제작 수칙
1. **Figma 의존성 탈피 및 원천 코드 추출**:
   - Figma 캡처 제공에 의존하지 않고, 대상 카드의 React 소스코드(`*.tsx`)의 마크업 구조와 CSS 스타일시트(`*.scss`/`*.css`) 소스를 직접 분석하여 DOM 위계 및 픽셀 규격(여백, 폰트 비율, 디바이스 반응형 구조)을 정밀 복원하여야 합니다.
2. **높이 제약 제거 및 세로 스크롤 원천 차단**:
   - 듀얼 미리보기 기동 시 모바일 뷰포트(`.phone-screen`)에 강제 스크롤바가 생기지 않도록 `max-height`를 완전히 배제하고 `overflow: visible;` 구조를 유지해야 합니다.
   * 이미지 영역(`IMAGE AREA`) 및 배너 등은 고정 높이값을 사용하지 말고 `aspect-ratio` 혹은 화면 너비 대비 상대비율(예: 모바일 220px 이하, PC 180px 이하)을 세팅하여 뷰포트 내 레이아웃이 찢어지지 않도록 최적화해야 합니다.
3. **정밀 정렬 및 컴포넌트 툴팁 제어**:
   - 수량 슬라이더의 `[사용수량]` 등 동적 툴팁은 절대적인 하드코딩 좌표값 대신, 프로그레스 바 영역에 맞물리는 상대 위치(예: `left: 2.8%; top: 0px; transform: translateX(-50%);`) 및 정확한 `z-index`를 결합하여 화면 배율에 맞춰 툴팁 핀이 정확히 수량 핸들(Thumb) 위에 위치하도록 세밀한 CSS 좌표 보정을 완료해야 합니다.
4. **C# 프로그램 및 PowerShell Script 기반 UTF-8 BOM 보존 동기화**:
   - Windows 로컬 실행 시 인코딩이 CP949로 오인되어 한글 텍스트 깨짐(외계어 표현)이 일어나는 문제를 방지하기 위해, 와이어프레임 파일(`db/wireframes/wf_*.js`) 수정 시 반드시 **UTF-8 with BOM (EF BB BF)** 인코딩 형식을 영구 보존해야 합니다.
   - **중요**: 단순히 텍스트를 쓸 때 인코딩이 깨진 상태로 저장되지 않도록, PowerShell이나 시스템 도구를 사용할 때는 반드시 `$text = [System.IO.File]::ReadAllText('...'); [System.IO.File]::WriteAllText('...', $text, [System.Text.Encoding]::UTF8)`와 같이 명시적 UTF-8 인코딩을 적용해 쓰기 작업을 거쳐야만 한글 깨짐을 원천 방지할 수 있습니다. 이를 준수하지 않으면 저장 시점에 텍스트가 유실되거나 붕괴합니다.
5. **실제 데이터 매핑 및 디자인 완성도**:
   - 탭 분류바, 2단 필터칩, 상품 브랜드 라벨, 할인율 배지 등 LFmall 상용 프론트 규격과 1:1 대칭을 이루도록 하여 심미성을 유지합니다.
6. **[NEW] 원천 코드 다중 버전(V1/V2) 최신성 교차 검증 규정**:
   - 컴포넌트 분석 시 `lf-ui` 내에 `planningTemplate` (구버전 V1) 외에 `planningTemplateV2` (신버전 V2) 폴더가 병렬로 존재하는지 반드시 먼저 확인해야 합니다. V2 폴더가 존재한다면 반드시 V2 컴포넌트(`TopType1.tsx`, `TopType2.tsx` 등)를 기준으로 원천 마크업과 스타일을 추출해야 합니다.
7. **[NEW] PC & Mobile 디바이스별 독립 듀얼 레이아웃 구현 규칙**:
   - PC 뷰포트(`desktop-mockup`)와 모바일 뷰포트(`phone-mockup`)의 프론트 구현이 서로 다른 구조(예: PC 가로 분할 vs 모바일 세로 3:4 오버레이)를 가지고 있다면, 단일 와이어프레임 파일 내에 두 가지 구조를 모두 작성하고, `.desktop-mockup` 및 `.phone-mockup` 스코프별 CSS를 사용하여 미디어를 분기 노출시켜야 합니다.
8. **[NEW] 텍스트 컬러 테마 및 글자 가독성 완전 대비 원칙**:
   - 배경 이미지 톤에 맞춰 텍스트 테마(전체 White 또는 전체 Black)를 완전 일관되게 단일 선택하여 통일해야 합니다. PC와 Mobile의 글자색이 다르게 들어가는 불일치를 절대 방지하십시오. 또한, Black 텍스트 테마일 때는 배경 사진 또한 매우 밝은 계열(예: 밝은 패션 스튜디오 백그라운드)의 이미지를 사용하여 모바일 오버레이 가독성을 영구 보장해야 합니다.
9. **[NEW] 글로벌 CSS 오버라이드 원천 방어 룰**:
   - 가이드 마스터 스타일시트(`style.css`)의 전역 태그 스타일(예: `h1 { color: var(--near-black) }`)에 의해 와이어프레임 내부의 의도된 색상이나 레이아웃이 침범당하지 않도록, 와이어프레임 내의 모든 자식 텍스트 노드(`h1`, `h2`, `p`, `a` 등)에는 명확하게 `color: inherit`, `font-size: inherit` 등을 부여하거나 정밀한 스코프 클래스명을 강제하여 스타일 오버라이드를 방지해야 합니다.
10. **[NEW] 다국어 텍스트 및 버튼 레이블 완전 동기화**:
    - PC와 Mobile 레이아웃에 노출되는 기간, 날짜 정보, 버튼의 영문/국문 텍스트(예: "BUTTON" vs "상품 바로가기") 등 모든 하드코딩 리터럴은 한 글자의 오차도 없이 양쪽 모두 완벽하게 동일하도록 동기화해야 합니다.
